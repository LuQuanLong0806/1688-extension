## 一、现有架构能力盘点

Pipeline 文档中列出的 7 个步骤所需的底层能力**全部已存在**，不需要从零开发任何核心模型调用。

| 能力层 | 现有模块 | 文件位置 | 状态 |

| 图片抠图 | ISNet ONNX 本地推理 | `server/services/remove-bg.js` | 可用 |

## 二、可行性分析

### 2.1 技术可行性 — 高（9/10）

**核心优势：**

- 所有 AI 调用均走 HTTP API（智谱/通义/腾讯云），本地 ONNX 模型已集成，无 GPU 硬性依赖
- 串行队列设计合理，"一次只处理一个商品"策略有效避免 API 限流
- 多 Key 轮换机制已成熟（智谱多Key、通义多Key、腾讯云多账户），Pipeline 直接复用
- SQLite 本地 + Turso 云同步架构已稳定运行
- 所有 Pipeline 步骤对应的服务函数均为已有模块，Pipeline 本质是**编排而非开发**

**风险评估：**

| 风险项 | 级别 | 说明 |
|--------|------|------|
| 单商品处理耗时 | 中 | 7步串行 ~18s/商品，100个商品 ≈ 30分钟。需前端可靠的进度反馈 |
| ONNX 模型内存占用 | 低 | ISNet + LaMa 同时加载约 300-500MB，可考虑懒加载/卸载 |
| PaddleOCR 进程稳定性 | 中 | 密集调用时需确保 OCR 子进程稳定，建议加进程健康检查 |
| 图片编解码质量损失 | 低 | 多步处理时每次编解码引入微小损失，可通过 pipeline 合并优化 |

### 2.2 业务可行性 — 高（8/10）

- `SKU > 6 跳过` 设计务实，避免复杂变体商品出错
- `automation_stage` 与 `status` 正交设计，不破坏现有发布逻辑
- 错误处理策略（单步失败 → 记录 → 跳过 → 继续）保证部分成功也能进入草稿箱
- 幂等性设计（已 processing 的跳过）避免重复处理

### 2.3 数据流可行性 — 中高（7/10）

新增 5 个字段（`automation_stage`、`automation_log`、`automation_issues`、`automation_started_at`、`automation_finished_at`）需要同步到 Turso 云端。需确保：

- 云端表自动补列逻辑覆盖新字段
- `saveProductToLocalAndCloud` 创建商品时写入默认值
- 网络异常时本地数据不丢失（先写本地再同步云端）

## 三、优化点

### 3.1 步骤内并行 — 节省 30-40% 处理时间

**现状：** 文档设计为 `商品串行 × 步骤串行`，7 步完全顺序执行。

**优化方案：** 质量检测通过后，识别无依赖关系的步骤并行执行。

```
商品A 优化后流程：

  Step 1: 质量检测（必须等结果，决定后续策略）
      ↓
  ┌─ Step 2: 去水印 ──────────┐
  │                            │  ← 并行执行
  └─ Step 3: 白底图生成 ───────┘
      ↓
  Step 4: 尺寸标注（依赖 Step 2 结果）
      ↓
  ┌─ Step 5: 分类推荐 ────────┐
  │                            │  ← 并行执行
  └─ Step 6: 图片上传 ImgBB ──┘
      ↓
  Step 7: 数据诊断 & 标记完成
```

**效果：** 单商品从 ~18s 降至 ~11-12s，批量处理提速约 35%。

### 3.2 智能步骤跳过 — 质量检测作为路由决策

**现状：** 质量检测结果仅记录日志，不决定后续步骤是否执行。

**优化方案：** 质量检测步骤输出结构化结果，驱动后续步骤的执行/跳过。

```
质量检测结果:
  has_watermark: false  → 跳过 Step 2 去水印
  has_chinese_text: true → 执行 Step 2（仅清理文字区域）
  background_simple: true → 跳过 Step 3 白底图
  quality_score: 95     → 无需额外处理
  has_size_info: true   → 执行 Step 4 尺寸标注
```

**收益：**
- 减少无效 API 调用（免费模型也有频率限制）
- 减少图片质量损失（每次 inpaint 都有微小质量损失）
- 加快处理速度（跳过步骤直接省时）

### 3.3 分类推荐增强 — 视觉 + 文本双通道交叉验证

**现状：** 分类推荐仅基于文本关键词提取 + 计分匹配。

**优化方案：** 结合 GLM-4V-Flash 视觉模型做"视觉分类"，与文本分类结果交叉验证。

```
Step 5 增强流程:

  1. GLM-4V-Flash 看图识别: "这是一个白色陶瓷马克杯，属于餐饮具"
  2. GLM-4.7-Flash 文本分析: 标题 + 属性 → 推荐 "杯子" 分类
  3. 交叉验证:
     - 一致 → 高置信度（0.9+），直接填入 custom_category
     - 不一致 → 低置信度（0.5），在 automation_issues 中标记:
       { "code": "category_conflict", "level": "warning",
         "message": "视觉识别为'杯子'，文本推荐为'家居'，需人工确认" }
```

**实现成本：** 视觉模型调用函数 `visionLLMRequest` 已有，只需在分类推荐步骤增加一次并行调用。

**适用场景：** 标题不准确的商品、无标题商品、跨品类商品（如"多功能收纳盒"既可以是家居也可以是办公）。

### 3.4 图片处理管道合并 — 减少质量损失累积

**现状：** 去水印 → 白底图分两步独立执行，每次独立编解码。

**优化方案：** 识别到需要多步处理的图片，合并为一个 sharp pipeline。

```
优化前（3次编解码）:
  原图 JPEG → LaMa inpaint → PNG → ISNet 抠图 → PNG → 合成白底 → JPEG

优化后（1次编解码）:
  原图 JPEG → 解码为 raw pixels
    → LaMa inpaint（在 raw 上操作）
    → ISNet 抠图（在 raw 上操作）
    → 合成白底
    → 一次编码为 JPEG
```

**技术基础：** `replace-bg-composite.js` 已有合成逻辑，`sharp` 支持链式 raw 操作。

### 3.5 队列优雅恢复 — 防服务重启误标失败

**现状：** 文档设计为"服务重启时 processing 状态标记为 failed"。

**优化方案：** 区分超时卡死和正常重启。

```
服务启动时队列恢复逻辑:

  1. 扫描所有 stage = 'processing' 的商品
  2. 检查 automation_started_at:
     - 超过 10 分钟 → 标记 failed（确实卡死了）
     - 未超过 10 分钟 → 重新加入队列头部，从断点继续
  3. 读取 automation_log 中最后完成的 step，从下一步开始
  4. 如果 log 为空（第一步就挂了）→ 从头开始处理
```

**收益：** 服务快速重启（如部署更新）时不会误杀正在处理的商品。

### 3.6 批量优先级排序 — 快速反馈体验

**现状：** 所有商品 FIFO 处理。

**优化方案：** 按预估处理速度排序。

```
用户选中 50 个商品批量自动化时:

  排序策略:
    1. 图片少的商品优先（1-2张图 vs 6张图，处理时间差 3-5 倍）
    2. 无水印商品优先（可跳过去水印步骤）
    3. 已有分类信息的靠后（跳过分类推荐步骤）

  前端展示:
    "已完成 5/50，预估剩余 12 分钟"
    已完成的商品即时出现在草稿箱
```

**收益：** 用户 30 秒内就能看到第一批处理结果，心理等待感大幅降低。

## 四、功能扩展建议

### 4.1 智能质检报告 — 人可读的质量分析

**场景：** 当前质量检测只输出 pass/fail + issues 列表，人工审核时信息不够直观。

**方案：** GLM-4V-Flash 输出结构化 + 自然语言质检报告。

```
GLM-4V-Flash 分析 3 张图片后输出:

{
  "quality_score": 78,
  "summary": "图片整体质量中等，第2张存在水印需要处理",
  "details": [
    { "image_index": 0, "score": 90, "issues": [],
      "note": "清晰度良好，无水印，背景偏暗" },
    { "image_index": 1, "score": 55, "issues": ["watermark", "blur"],
      "note": "右下角有1688水印，轻微模糊，建议去水印并增强" },
    { "image_index": 2, "score": 88, "issues": [],
      "note": "包含尺寸信息(60×40cm)，可用于尺寸标注" }
  ]
}
```

**实现：** 在质量检测步骤的 prompt 中要求结构化输出，存入 `automation_log.steps[0].result`。草稿箱详情弹窗中直接展示。

### 4.2 视觉属性提取 — 一次调用完成多任务

**场景：** 分类推荐只填了 category，但商品还有很多可自动提取的属性。

**方案：** 在视觉检测步骤中一次提取多个维度。

```
GLM-4V-Flash 一图多任务输出:

{
  "category": "杯子",
  "attributes": {
    "color": ["白色", "蓝色"],
    "material": "陶瓷",
    "style": "简约",
    "scene": ["家用", "办公"],
    "shape": "圆柱形",
    "pattern": "纯色"
  },
  "quality_note": "产品居中展示，光线均匀"
}
```

**应用场景：**
- 自动生成商品标题关键词
- 丰富搜索标签，提升搜索命中率
- 在待发布页面按属性分组（如"所有白色商品"）
- 自动匹配目标平台属性模板

**实现成本：** 只需修改视觉检测步骤的 prompt 模板，一次 API 调用完成。

### 4.3 智能重试 Agent — 自动修复可恢复错误

**场景：** 当前设计是"单步失败 → 记录 → 跳过"。部分错误是可恢复的（网络超时、API 限流）。

**方案：** 增加自动重试层，区分可恢复错误和不可恢复错误。

```
错误分类与处理策略:

  可恢复错误（自动重试1次）:
    - 网络超时 / 连接重置    → 等 2s 后重试
    - API 限流 (429)         → 等 5s 后用下一个 Key 重试
    - ComfyUI 服务暂时不可用  → 等 3s 后重试
    - PaddleOCR 进程重启中   → 等 5s 后重试

  不可恢复错误（直接跳过）:
    - 模型返回格式错误       → 记录错误，跳过该步骤
    - 图片解码失败           → 记录错误，跳过该步骤
    - 商品数据不完整         → 标记 failed

  降级策略:
    - LaMa inpaint 失败 → 降级到 ComfyUI inpaint
    - ISNet 抠图失败   → 降级到 ComfyUI Rembg
    - GLM-4V 检测失败  → 跳过质量检测，直接处理（保守策略）
```

**实现：** 在 Pipeline 每个步骤外包装一个 `retryWrapper(fn, maxRetries, fallbackFn)` 函数。

### 4.4 ComfyUI 高级图片工作流 — 场景图生成

**场景：** 白底图适合部分平台，但某些平台/品类需要场景图（商品放在桌面上、手持展示等）。

**方案：** 利用已集成的 ComfyUI 设计高级工作流。

```
ComfyUI 工作流（可配置）:

  工作流 A — "白底商品图":
    原图 → ISNet 抠图 → 合成白底 → 加自然阴影 → 输出

  工作流 B — "场景展示图":
    原图 → ISNet 抠图 → CogView 生成场景 → 合成 → 调色 → 输出
    场景预设: "木质桌面" / "大理石台面" / "简约家居"

  工作流 C — "多角度展示":
    原图 → 抠图 → 自动生成正面/侧面/俯视三个角度的展示图
```

**技术基础：** `comfyui-inpaint.js` 已有 ComfyUI API 调用封装，只需设计新的工作流 JSON。

**应用：** 在 Pipeline Step 3 中增加"图片风格"选项（白底/场景/多角度），用户可在触发批量自动化时选择。

### 4.5 本地模型预处理筛选 — 节省 API 调用

**场景：** 并非所有图片都需要完整的 7 步处理，先用本地模型做轻量判断。

**方案：** 利用本地 ONNX 模型快速推理，筛选出真正需要处理的图片。

```
本地预处理（ISNet 推理 ~200ms/张）:

  1. ISNet 推理得到前景 mask
  2. 分析 mask 的统计特征:
     - 前景占比 > 90% → 背景极简，可能不需要白底图
     - 前景边缘复杂度低 → 物体形状简单，抠图容易
     - 前景占比 < 30%  → 物体很小，白底图可能效果不好
  3. 输出处理建议:
     { "need_bg_removal": true, "bg_complexity": 0.7, "recommended": true }
```

**收益：** 对背景简单的商品跳过白底图步骤，节省 ~5s/商品。

### 4.6 自动标题优化 — 利用 AI 生成平台适配标题

**场景：** 1688 原始标题直接搬到其他平台往往水土不服（堆砌关键词、格式不统一）。

**方案：** 在分类推荐步骤之后增加"标题优化"步骤。

```
标题优化流程:

  输入: 原始标题 + 分类 + 视觉提取的属性
  处理: GLM-4.7-Flash 根据目标平台规则重写标题

  示例:
    原始: "北欧简约陶瓷马克杯大容量办公室水杯定制logo家用牛奶杯早餐杯"
    优化: "北欧简约陶瓷马克杯 | 大容量500ml | 白色/蓝色 | 家用办公"
```

**应用：** 在草稿箱详情弹窗中展示原标题 vs 优化标题对比，用户一键采纳或手动调整。

### 7.2 核心差异对照表

| 维度 | 原方案（v1.0） | 优化方案 |
|------|---------------|----------|
| **执行模型** | 7步完全串行 | 质量检测后分组并行（2+2+1+2） |
| **单商品耗时** | ~18s | ~11s（无跳过）；~6-8s（有步骤跳过） |
| **质量检测角色** | 仅记录，不影响后续 | 路由决策，驱动步骤跳过/执行 |

| **白底图策略** | 所有图片都执行 | 背景简单 → 跳过；复杂 → 执行 |

| **错误处理** | 失败 → 记录 → 跳过 | 失败 → 自动重试1次 → 降级备选 → 跳过 |

| **批量排序** | FIFO 先进先出 | 按图片数量/复杂度排序，快速商品优先 |
| **图片处理** | 每步独立编解码 | 多步合并为一个 sharp pipeline |
| **质检输出** | issues 列表 | 人可读报告（自然语言 + 结构化数据） |
| **属性提取** | 仅分类 | 分类 + 颜色/材质/风格/场景等多维属性 |

### 7.3 错误处理对比

```
原方案：
  任何错误 → 记录到 log → 跳过该步骤 → 继续下一步

优化方案：
  任何错误 → 判断是否可恢复？
    ├─ 可恢复（网络超时/限流/服务暂时不可用）
    │    → 等待 + 自动重试1次
    │    → 重试仍失败 → 判断是否有降级方案
    │       ├─ 有降级（LaMa→ComfyUI / ISNet→Rembg）
    │       │    → 执行降级方案
    │       │    → 降级成功 → 继续
    │       │    → 降级失败 → 记录 → 跳过
    │       └─ 无降级 → 记录 → 跳过
    └─ 不可恢复（格式错误/数据损坏）
         → 记录 → 跳过
```

### 8.2 Agent 的核心价值：模型分级调度

**原方案的分类推荐只有一个入口：** GLM-4.7-Flash 文本分析 → 关键词匹配 → 互斥组过滤。

**Agent 的思路完全不同：** 它不是一个固定流程，而是一个"会思考的路由器"——根据商品的复杂度，自动选择用哪个模型、做几层分析。

```
传统流程（原方案）:
  商品 → [固定步骤1] → [固定步骤2] → ... → [固定步骤7] → 结果
  不管商品是简单还是复杂，都走同样的路

Agent 模式:
  商品 → Agent 分析复杂度 → 自适应选择模型和步骤
    简单商品: 1-2个免费模型搞定（0成本）
    复杂商品: 升级到付费模型做深度分析（低成本）
    结果不确定: 多模型交叉验证（中等成本）
```

### 8.8 Agent 模式实现要点

```
核心逻辑（伪代码）:

  async function agentAnalyze(product) {
    // ── Layer 1: 免费预判 ──
    const freeVision = await glm4vFlash(product.images);
    // GLM-4V: 看图输出分类 + 水印检测 + 质量评估

    const freeText = await glm47Flash(product.title, product.attributes);
    // GLM-4.7: 文本提取关键词 → 匹配分类库

    // 交叉验证
    if (agree(freeVision.category, freeText.category)) {
      if (confidence >= 0.8) {
        return { category: freeText.category, confidence, cost: 0 };
        // 80% 的商品在这里结束，0成本
      }
    }

    // ── Layer 2: 低价精分析 ──
    const deepAnalysis = await qwen36Flash(product.images[0]);
    // Qwen3.6-Flash: 深度图片分析，成本≈0.001元

    const merged = crossValidate(freeVision, freeText, deepAnalysis);
    if (merged.confidence >= 0.85) {
      return { ...merged, cost: 0.001 };
      // 15% 的商品在这里结束
    }

    // ── Layer 3: 高精度兜底 ──
    const precise = await qwen37Plus(product.images);
    const hunyuanCheck = await hunyuanLite(product.title);
    // 第三视角验证

    const final = vote([freeVision, freeText, deepAnalysis, precise, hunyuanCheck]);
    return { ...final, cost: 0.004, needsReview: final.confidence < 0.8 };
    // 5% 的商品到这里，标记需人工确认
  }
```

**与现有代码的对接：**
- `visionLLMRequest()` → Layer 1 视觉分析（已有）
- `categoryLLMRequest()` → Layer 1 文本分析（已有）
- `qwenVlRequest()` → Layer 2/3 深度分析（已有）
- `runLLMChain()` → 降级链自动切换（已有）
- 多 Key 轮换 + 冷却机制 → 直接复用（已有）

**不需要新写任何模型调用函数，只需要在 Pipeline 层加一个调度器。**

## 九、总结

这个 Pipeline 不是从零构建新系统，而是对现有 8+ 个独立服务模块的**编排整合**。最大的工程优势是"所有零件已备齐，只需组装"。

真正的工程重心不在模型调用本身，而在：

1. **错误处理的健壮性** — 每个步骤可能出错的场景远多于正常场景
2. **前端状态同步的流畅度** — SSE 实时反馈、草稿箱即时刷新、批量操作的视觉反馈
3. **可观测性** — automation_log 的结构化设计决定了后续排查问题的效率
4. **Agent 调度的智能性** — 用免费模型做 80% 的工作，只在必要时花极低成本升级

**Agent 集成的核心结论：**

- 分类准确率可从 ~75-80% 提升到 ~92-95%
- 100 个商品的额外模型成本不超过 0.1 元（Qwen3.6-Flash 极低价）
- 所有模型调用函数已存在于 `providers.js`，无需新写任何 API 对接
- Agent 调度器本质是一个 "置信度门控 + 模型升级" 的路由逻辑，实现简单

优化空间最大的是"Agent 自适应调度 + 智能步骤跳过"两个方向，投入产出比最高。建议 P0 完成后立即实施 P1 优化，其中 Agent 调度器应作为 P1 的核心。
